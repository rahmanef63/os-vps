## What shipped this session

**`b4b90c5` — fix(host): stream fs/upload with running 200MiB cap (P0).**
`/api/v1/fs/upload` used `req.formData()` + `arrayBuffer()`, which buffers every
multipart part fully into RAM before the handler runs — a large authenticated
upload OOM-kills the host process that *is* the cockpit. Replaced with a
hand-rolled streaming parser:
- `lib/host/multipart.ts` — consumes `req.body` as a stream, enforces a running
  total-bytes cap, yields each part as a byte stream.
- `lib/host/fs-upload.ts` — spools each part to a `.tmp` via `createWriteStream`
  (backpressure) + atomic rename, always within `lib/host` realpath/write-root
  bounds. (`uploadInto` moved here to keep `fs.ts` under the 200-line rule;
  `browser/save-shot` consumer unaffected — imports via the `@/lib/host` barrel.)
- `next.config.mjs` — `proxyClientMaxBodySize` 500mb → 256mb.

**`4ddc70f` — fix(host): cap raw multipart bytes + add parser tests.**
A follow-up QA pass found the cap only counted *part-body* bytes, so a malicious
oversized **preamble** or **header block** (no boundary / no CRLFCRLF) could
still grow the read buffer unbounded and OOM. Fixed by capping **raw** bytes
pulled from the stream (covers preamble + headers + bodies uniformly). Added
`lib/host/multipart.test.ts` — 6 tests: happy path, body cap, preamble bypass,
header bypass, boundary parsing.

## ⚠️ Concurrent-session incident (the lesson)

Two Claude sessions ran fix-loops on the **same working tree at the same time**.
os-vps is a single checkout — one tree, one `HEAD`, one git index — so the two
sessions collided **three times**:
1. The other session's `git reset --hard` / `git stash` wiped this session's
   uncommitted work mid-flight (recovered from an out-of-band backup).
2. Same again on a later wave.
3. **Shared index sweep:** between this session's `git status` and `git add`, the
   other session `git add`-ed 6 of its files; this session's `git commit` swept
   them into `4ddc70f` (so `audit.ts`, `audit.test.ts`, `exec-filter.test.ts`,
   `paths.test.ts`, `scripts/e2e/smoke.test.ts`, `INSTALL.md` are committed under
   a `fix(host): …multipart…` message). **Content is correct + green**, only the
   commit attribution is mixed. It was left as-is on purpose: un-bundling needs a
   history rewrite of the shared `HEAD`, which would just cause more collisions.

**Rule going forward: only one Claude session edits os-vps at a time.** Before a
long fix-loop, `git log --oneline -10` to check for unexpected recent commits =
another session is active → stop and run solo.
