os-rr Cloud API

The single boundary between the os-rr desktop and your VPS. Everything the OS does — files, processes, renders, app lifecycle — is one of these calls. Base URL https://<vps>/api/v1. JSON in, JSON out, Bearer auth, SSE for progress.

Overview

Run the os-rr daemon on your VPS and the desktop's HttpAdapter maps every UI action to the endpoints below. In the prototype, the identical MockAdapter simulates them in-browser — switch in Settings → Server.

ConceptDetail
Base{baseUrl}/api/v1
AuthAuthorization: Bearer <token> on every request (SSE uses ?access_token=)
Contentapplication/json unless multipart (upload) or binary (read/download)
PathsAbsolute within the user root, e.g. /Media/clip.mp4
Errors{ "error": { "code", "message", "detail?" } } with matching HTTP status
ProgressServer-Sent Events for render / stats / exec output; WebSocket for interactive PTY

Filesystem

CRUD over the VPS disk. Drives the Files app, media import, and app packages.

GET/fs/list
QueryType
pathstringrequired
200
{
  "path": "/Media",
  "entries": [
    { "name":"intro.mp4", "kind":"file", "size":50331648, "ext":"mp4", "mime":"video/mp4", "mtime":1736890000 },
    { "name":"raw", "kind":"dir" }
  ]
}
GET/fs/read  GET/fs/stat  GET/fs/download

read streams bytes (supports Range); stat returns one entry; download sets Content-Disposition: attachment for the browser. All take ?path=.

POST/fs/write

Create or overwrite a text file.

{ "path":"/Documents/notes.md", "content":"# hi", "encoding":"utf8" }
POST/fs/upload

multipart/form-data — fields path + repeated files. For very large media, the daemon may instead return a presigned PUT URL. Returns the created entries.

// FormData: path=/Media, files=[File, File]
{ "uploaded": [ {"name":"shot.png","size":4200000} ] }
POST/fs/mkdir  POST/fs/move  POST/fs/copy  DELETE/fs/delete

mkdir {path} · move {from,to} (also rename) · copy {from,to} · delete {path, toTrash?}. Each returns the affected entry or {ok:true}.

GET/fs/usage
{ "used":289000000000, "total":460000000000 } // bytes

Exec & Terminal

Run scripts and apps; back the Terminal with an interactive shell.

POST/exec/run

Spawn a process. Returns a pid immediately; stream output via SSE. Pass wait:true for synchronous {exitCode,stdout,stderr}.

{ "cmd":"python", "args":["scraper.py"], "cwd":"/apps", "env":{} }
→ { "pid":418, "state":"running" }
SSE/exec/stream?pid=418
event: {"type":"stdout","data":"scraping…"}
event: {"type":"exit","code":0}
WS/exec/pty  POST/exec/kill

Bidirectional WebSocket for an interactive shell: client sends {input} / {resize:{cols,rows}}, server sends {output}. kill {pid} terminates a process.

GET/render/status?jobId=…  SSE/render/stream?jobId=…
{
  "jobId":"job_a1b2c3", "state":"rendering",  // queued|rendering|done|error
  "progress":0.62, "frame":186, "totalFrames":300,

POST/render/cancel  GET/render/jobs

cancel {jobId} stops a job; jobs lists active/recent renders.

Apps

Each app is a folder under /apps/<slug>/ with a manifest.json. The daemon reads manifests, serves HTML entry points, and runs other runtimes in a sandbox.

GET/apps  GET/apps/:slug/manifest
[ { "id":"color-picker", "name":"Color Picker", "slug":"color-picker",
     "runtime":"html", "entry":"index.html", "icon":{"glyph":"grid"} } ]
POST/apps

Create an app folder from a manifest (Create App flow). Optionally include initial files.

{ "manifest": { "name":"My App", "runtime":"html", "entry":"index.html" } }
POST/apps/:slug/start  POST/apps/:slug/stop  DELETE/apps/:slug

start launches the app — HTML apps return a url the desktop loads in a sandboxed iframe; runtime apps return a pid. GET /apps/:slug/serve/* is the static file server for an HTML app's assets.

Streaming

Long-running operations push updates instead of being polled.

TransportUsed byWhy
SSErender progress, sys stats, exec outputOne-way, auto-reconnect, plain HTTP. Auth via ?access_token=.
WSinteractive terminal (PTY)Bidirectional — keystrokes up, output down, resize events.
Clients should treat SSE as best-effort: if the stream drops, fall back to polling the matching GET …/status endpoint. The prototype's MockAdapter emulates both with timers.

App manifest

/apps/<slug>/manifest.json — the contract for a user-created app.

{
  "name": "My App",
  "slug": "my-app",
  "runtime": "html",            // html | node | python | shell
  "entry": "index.html",        // served / executed
  "icon": { "glyph":"grid", "color":"#7a5cff" },
  "window": { "width":900, "height":600 },
  "env": {},
  "permissions": [ "fs:/Media", "net" ]
}
This contract is implemented as code in js/api.jsx — HttpAdapter (live) and MockAdapter (simulated) share one shape, so the desktop is wired and ready: stand up a daemon that answers these routes, flip Settings → Server → Live, and point the base URL at it.
