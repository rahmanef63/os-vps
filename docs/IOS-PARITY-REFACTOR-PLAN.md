# iOS Parity Refactor Plan — os-vps feature slices vs the Apple mock

> **Goal:** bring every os-vps **feature slice's iOS (iPhone) surface** to parity with the
> shared Apple iOS shell in `mock-os/Apple-clone-app/*` — grouped inset lists, liquid-glass
> chrome, large-title nav, iMessage-style bubbles, iOS controls — **without** regressing the
> **android / windows / dashboard / macOS** shells and **without** breaking os-vps's essence
> as a headless-VPS cockpit.
>
> **Scope:** presentation only. No data layer, no host-I/O, no window-store fork, no new deps.
> Built from a 19-agent audit (2026-07-15): 3 SSOT extractors + 1 synth read the mock; 15 diff
> agents (one per slice group) compared each slice against the SSOT. Every metric below is
> **verified against `theme.js` / `config.js` / `support.js` / the `.dc.html` screens** — see
> the per-slice `_ref`.

## 0. Hard constraints (read first)

Non-negotiable. Every step in §4 is checked against these.

1. **VPS essence — never fabricate hardware.** No status-bar clock / wifi / battery / cellular /
   signal / brightness anywhere. The mock draws all of them on every screen; we reserve the notch
   zone (`--sai-top`) and paint **nothing**. The Dynamic Island stays a **functional live-activity
   pill** (shows only when an app reports an activity), never a decorative idle black notch.
   Control Center stays **real-toggles-only** (appearance/server/focus/spotlight/assistant/windows).
2. **File-explorer DATA layer is sealed.** Every host-I/O symbol stays routed through `lib/host.ts`
   (`use-files*`, zip/upload/inspector untouched). The files-manager work is **presentation-only**;
   any iOS "search" must filter already-loaded entries client-side, never issue a host query.
3. **One window store.** Shells stay chrome-only; never fork `appshell/lib/store.ts`.
4. **os-vps rules stay in force:** shadcn primitives only, theme tokens not raw hex (fixed
   *category* colors — folder `#57b3ff`, switch-ON `#30d158`, audio gradient — are the accepted
   exception), ≤200 lines/file, barrel-only cross-slice imports (`@/features/<slug>`), no new deps.

### 0.1 The two seams — how android/windows/dashboard stay byte-unchanged

Every iOS change rides **one of two already-live seams**. This is the whole safety story:

| Seam | Use for | Why the other shells are safe |
|---|---|---|
| **`useActiveShell().id === 'ios'`** (React) | structural / markup branches (large title, sheet-vs-dialog, composer pill, nav restructure) | the `else` path renders **today's exact markup**; android/windows/dashboard/macOS take it verbatim |
| **`[data-shell="ios"] …`** (CSS) | pixel/color/token deltas on **shared** primitives (`Switch`, `Segmented`, `.glass` edge, icon shadow) | android matches `[data-shell="android"]`, etc. → **zero** computed-style delta on their trees |

**The single most important rule: gate on `id === 'ios'`, NOT `surface === 'mobile'`.** Android is
*also* `surface === 'mobile'` and renders its own `android-shell.tsx`. Keying on `surface` would
silently restyle Android. Every step below keys on the exact string `'ios'`.

**Bonus — already partitioned by mount.** `mobile-shell.tsx` and its whole tree (`mobile-home*`,
`mobile-switcher`, `home-indicator`, `dynamic-island`, `mobile-notifications`) are the `render:` of
**exactly one** ShellDescriptor (`id:"ios"`, `desktop.tsx:211`). No other shell imports them, so
editing these files is invisible to the others **with no branch at all**. Shared primitives
(`components/ui/*`, `WindowContent`-mounted app bodies) are the only things that need a gate.

### 1.5 Other reusable patterns
- **Search field:** `--fill` pill, radius ~10–12, leading search glyph `15×15 --label2` + placeholder. (Safari swaps for a centered URL pill `h38 r11`.)
- **Sheet + detents (§3.3):** rounded-top `20px`, **SOLID `--elev` — never glass**, grabber `36×5 r2.5 --label3` top-center, dimmed scrim, medium/large detents.
- **Segmented:** track `--fill r9 pad2`, option `14/500 r7 pad7×4`, selected = raised white `--card` pill + shadow.
- **Switch:** `47×29 r15`, knob `25×25` white, ON `#30d158` / OFF `--fill2`.
- **Buttons:** filled/prominent `--tint` bg + white 600 · tinted/plain `--tint` text · icon FAB `30×30` circle `--tint`.
- **Large-title collapse:** at rest 32/800; on scroll → centered inline 22/700 inside the 46px nav, which gains glass + .5px hairline.
- **Empty state:** centered column, `--label2` 15px, `padding:70px 20px`, "No Items".

### 1.6 Golden Liquid-Glass rules
1. **Glass = navigation/chrome layer ONLY** (status/nav/tab/toolbar/sidebar/menu/dock). `--glass-bg` + `backdrop-filter:blur(26px) saturate(180%)` + `--glass-stroke` outer + `--glass-hi` inner.
2. **Never glass on content** — cards, rows, bubbles, sheets, list tiles = **solid**. Content scrolls *under* chrome; it never *is* glass.
3. **Never glass-on-glass.** One glass plane at a time; a menu over a glass bar renders on solid `--elev`.
4. **Text always on solid** (card/row/bubble, or the glass's own tinted fill) — never on raw wallpaper blur.
5. **Tint primary/interactive only** — nav buttons, links, selected tab/segment, send FAB, big-stat. Semantic system colors (green switch, red destructive) are separate from the user tint.

*Under Reduce Transparency all glass collapses to solid (blur 0, highlight off) — layout must stay correct with zero glass.*

## 3. Cross-cutting systemic fixes (do these ONCE, they pay off everywhere)

Three issues recur across many slices. Fixing them at the shared layer (iOS-scoped) is higher-leverage
than per-slice patches — **do them before the per-slice passes**.

### S1 — iOS-scope the shared `Switch` + `Segmented` controls `[HIGH]`
`components/ui/switch.tsx` ON = `bg-primary` (user accent); `components/ui/segmented.tsx` selected =
`bg-primary text-primary-foreground` (tint-filled pill). iOS wants **switch ON = system green
`#30d158`** and **segmented selected = raised *white* `--card` pill on a `--fill` track** (Golden
Rule 5: tint is not a fill). **This is a correctness boundary, not cosmetics** — Android (Material)
and Windows (Fluent) *want* accent-colored controls, so a global retune would **regress them**. Fix
via `[data-shell="ios"]` CSS + inert `data-slot`/`data-selected` attrs on the primitives (render-inert
for all shells). Unblocks shell-settings, create-app, app-store, media-picker, editors-misc at once.

### S2 — Fix the Reduce-Transparency glass bypass `[HIGH]`
Control Center (`backdrop-blur-xl`), Notification Center (inline `blur(28px)`), Dynamic Island
(`backdrop-blur`), browser chrome, and code-editor panels all use **fixed Tailwind/inline blur** that
**bypasses `var(--blur)`** — so glass never collapses when the user enables Reduce Transparency (an
a11y content-behavior, SSOT §1.7). Swap each to the repo `.glass` helper (which reads `var(--blur)`
and `.reduce-glass` already zeroes). ~1 line each; neutral-positive for Android CC, so no branch needed.

### S3 — Add the missing liquid-glass tokens `[MED]`
The mock's glass has an **inner top highlight + outer hairline** we don't model: add `--glass-hi` +
`--glass-stroke` (light/dark) and a `--grouped` page token (`#f2f2f7` / `#000`). Then an iOS-scoped
`[data-shell="ios"] .glass{ box-shadow:inset 0 .5px 0 var(--glass-hi); border-color:var(--glass-stroke) }`
(or opt-in `.glass-ios`) gives CC/NC/Spotlight/dock/nav the real liquid-glass edge. Additive tokens;
base `.glass` untouched. **Caveat:** Control Center's shadcn Sheet portals to `<body>` (outside
`[data-shell]`), so apply the edge class directly under the `id==='ios'` gate, not via a descendant selector.

## 6. Consolidated essence skips — the do-NOT-build list

Every agent independently confirmed these. They are **deliberate**, not gaps — ratify with a
`ponytail:` comment where the code makes the choice:

1. **No status bar.** Clock / wifi / battery / cellular / signal / brightness are fabricated hardware.
   Reserve the notch zone (`--sai-top`), paint nothing. *(mobile-home.tsx already documents this.)*
2. **Dynamic Island stays functional-only** — a live-activity pill that renders when an app reports an
   activity, never a decorative idle black notch pill.
3. **Control Center = real toggles only** — no brightness/volume/wifi/airplane/AirDrop modules (no
   such hardware on a headless VPS).
4. **No cross-app nav contract.** The Files floating tab bar and per-app nav trailing actions
   (share/compose/trash) need a shell↔app tab/action contract; os-vps apps surface actions in their
   own in-content toolbars. Skip *(system-monitor's optional nav-action slot is the one place to
   revisit if wanted)*.
5. **Files "search" filters loaded entries client-side** — never a dead cosmetic pill, never a new
   host query. The DATA layer (`lib/host.ts`, `use-files*`, zip/upload/inspector) is sealed.
6. **Don't restructure editor canvases** — Konva stage, reel timeline/preview, xterm PTY output stay
   as-is; only their surrounding chrome aligns.
7. **Keep the AA-contrast departures** — `SettingsActionRow` uses `--info`/`--destructive-text` (AA-safe)
   not the raw iOS tint-as-text (~3:1); the accent-swatch keeps its `Check` glyph (a11y over pixel
   parity). Both are *more* correct than the mock.
8. **Never globally retune Switch/Segmented** — Android (Material) and Windows (Fluent) want
   accent-colored controls; the iOS look is `[data-shell="ios"]`-scoped only.
9. **Consumer apps stay out of scope.** The mock's Mail / Maps / Messages / Music / Photos are
   portfolio apps a VPS cockpit doesn't have — we harvest their *chrome patterns* (bubbles, now-playing,
   grid tiles, address bar) for our real apps, never port the apps.
